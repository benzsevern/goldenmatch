# GoldenMatch v1.0.0 Design Spec

## Goal

Ship GoldenMatch 1.0 with three headline features (in-context LLM clustering, uncertainty scores, multi-party PPRL) plus CI/CD and API stability guarantees. The release drops the "Beta" label and commits to semver.

## Approach

Four sequential phases, each independently shippable:

- **Phase 0 (v0.4.0):** CI/CD pipeline + API stability audit
- **Phase 1 (v0.5.0):** In-context LLM clustering + uncertainty scores
- **Phase 2 (v0.6.0):** Privacy-preserving record linkage (bloom filter improvements + multi-party SMC)
- **Phase 3 (v1.0.0):** API freeze, documentation sweep, 1.0 release

CI/CD lands first so every subsequent feature ships with automated test coverage. Breaking API changes happen in Phase 0 while still in 0.x. PPRL gets the most runway as the highest-risk feature.

## Phase 0: CI/CD + API Stability Audit (v0.4.0)

### CI/CD Pipeline

GitHub Actions workflow on every push and PR:

- `pytest --tb=short --ignore=tests/test_db.py --ignore=tests/test_reconcile.py`
- Matrix: Python 3.11, 3.12, 3.13 on ubuntu-latest
- Ruff lint check as a separate job
- Build status badge in README
- Performance regression baseline: capture key benchmark times (1K, 10K exact+fuzzy) and fail CI if >2x slower

### API Stability Audit

- Review every public function signature in `core/`, `cli/`, `config/schemas.py`
- Audit package-level `__init__.py` exports -- define the public import surface explicitly
- Identify anything that should change before the 1.0 contract locks it in (parameter names, return types, config keys)
- Make breaking changes now, document in CHANGELOG.md
- Add `CHANGELOG.md` with Keep a Changelog format, retroactive entries for 0.3.0 and 0.3.1
- Add `py.typed` marker for PEP 561 type checker support

### Versioning Policy

After 1.0: semver strictly. Public API surface:

- CLI commands and flags
- Config YAML schema
- `core/` function signatures
- REST API endpoints
- MCP tools
- Package-level imports (`from goldenmatch.core.X import Y`)

Internal modules (anything prefixed `_`) are not part of the contract.

### Test Strategy

- All existing 855+ tests run in CI
- Add a smoke test job that installs from built wheel and runs `goldenmatch demo`

## Phase 1: In-Context LLM Clustering + Uncertainty Scores (v0.5.0)

### In-Context LLM Clustering

**New module:** `core/llm_cluster.py`

**Flow:**

1. Traditional pipeline runs as normal through blocking + scoring
2. Borderline pairs (0.75-0.95) are collected -- same as today
3. Instead of pairwise LLM calls, borderline records are grouped into blocks of 50-100
4. Each block is sent as a single prompt: "Here are N records. Group them into clusters of duplicates. Return cluster assignments."
5. LLM returns structured JSON output
6. Synthetic pair_scores are generated from LLM results (see integration below)
7. Results merge back into the pipeline's Union-Find

**Prompt design:** Records formatted as a numbered table (ID, key fields). Uses OpenAI JSON mode / structured outputs or Claude tool use to ensure valid JSON. If structured output is unavailable, parse with fallback regex extraction. System prompt includes domain context if a domain pack is active.

**Block formation:** Borderline pairs form a graph. Connected components of that graph become LLM blocks.

- Components exceeding `cluster_max_size`: split by removing edges below the median score until component fragments into sub-components under the limit. Deterministic, O(E log E).
- Components under `cluster_min_size`: fall back to pairwise LLM scoring (existing `llm_score_pairs`).

**Integration with cluster contract:**

The existing `build_clusters()` requires `pair_scores` for every cluster (used by `unmerge_record()`, `confidence`, `bottleneck_pair`, lineage, and explainability). LLM clustering produces group assignments, not pair scores. To bridge this:

- For each LLM cluster, synthesize pair_scores: every pair within the cluster gets `score = llm_confidence` for that cluster. This makes unmerge, confidence, and lineage work without code changes.
- Add `cluster["origin"] = "llm"` flag so downstream consumers can distinguish LLM-formed clusters from fuzzy-formed ones.
- Explainability for LLM clusters: `explain_pair_nl()` notes "Grouped by LLM with confidence X" rather than per-field breakdown.

**Config:**

```yaml
llm_scorer:
  enabled: true
  mode: cluster          # "pairwise" (legacy) or "cluster" (new)
  cluster_max_size: 100  # ignored when mode: pairwise
  cluster_min_size: 5    # ignored when mode: pairwise
```

Updated Pydantic model (`LLMScorerConfig`):
- Add `mode: Literal["pairwise", "cluster"] = "pairwise"`
- Add `cluster_max_size: int = 100`
- Add `cluster_min_size: int = 5`
- Validator: reject `cluster_max_size` / `cluster_min_size` when `mode == "pairwise"`

**Budget integration:** Same `BudgetTracker` -- each block call tracked by tokens/cost. Degradation chain: cluster mode -> pairwise mode -> stop. Budget exhaustion triggers graceful fallback, not hard failure.

**Fallback on LLM error:** If LLM returns malformed JSON or times out for a specific block, fall back to pairwise scoring for that block only. Pairwise fallback respects the budget tracker. Log a warning with block size and error.

### Uncertainty Scores

The LLM returns confidence alongside each cluster assignment:

```json
{
  "clusters": [
    {"members": [0, 3, 7], "confidence": 0.92},
    {"members": [1, 4], "confidence": 0.71}
  ],
  "singletons": [8, 9]
}
```

Confidence integration:

- `cluster["llm_confidence"]` stored in clusters dict (None for non-LLM clusters)
- For LLM-only clusters: `confidence = llm_confidence` (no blending -- fuzzy confidence is meaningless since pairs were scored by LLM)
- For hybrid clusters (some pairs from fuzzy, some from LLM): `confidence = min(fuzzy_confidence, llm_confidence)` (conservative -- weakest signal dominates)
- Clusters with `llm_confidence < 0.7` auto-flagged for human review in the review queue
- Surfaced in `goldenmatch evaluate` output, REST API `/clusters/{id}`, MCP `get_cluster` tool

### Test Strategy

- Mock LLM responses with fixture JSON files -- no live API calls in CI
- Test block formation: graph splitting, min/max size boundaries, edge cases (empty blocks, single-record blocks)
- Test pair_score synthesis: verify unmerge_record works on LLM clusters
- Test fallback: malformed JSON -> pairwise, budget exhaustion -> graceful stop
- Test confidence integration: LLM-only, hybrid, and fuzzy-only clusters
- Benchmark: run on Abt-Buy with mocked LLM to measure integration correctness (not accuracy -- that needs live API)

## Phase 2: Privacy-Preserving Record Linkage (v0.6.0)

### Phase 2a: Better Bloom Filter Accuracy

Current PPRL uses basic CLK bloom filters with SHA-256. Known weaknesses: common values produce predictable bit patterns, vulnerable to frequency attacks, lose signal on short strings.

**Improvements:**

- **Frequency-aware bit sampling** -- weight n-gram hash positions by inverse frequency. Common n-grams get fewer bits, rare n-grams get more. Reduces false positives from common names.
- **Balanced bloom filters** -- pad short strings with random salt before hashing so all filters have similar density. Prevents length-based inference.
- **Field-level salting** -- different HMAC key per field (name, address, DOB) so cross-field correlation attacks don't work.
- **Configurable security parameter:**

| Level | n-gram | Hash functions | Filter size | Salting |
|-------|--------|---------------|-------------|---------|
| `standard` | bigram (2) | 20 | 512 bits | single key |
| `high` | bigram (2) | 30 | 1024 bits | per-field HMAC |
| `paranoid` | trigram (3) | 40 | 2048 bits | per-field HMAC + balanced padding |

These live in existing `utils/transforms.py` (bloom filter transform) and `core/scorer.py` (dice/jaccard scoring). No new modules.

**Accuracy targets:** Measure on a synthetic name+address dataset with known ground truth. Current baseline false positive rate TBD during implementation. Targets: `standard` <= 5% FPR, `high` <= 2% FPR, `paranoid` <= 1% FPR at 0.85 dice threshold.

### Phase 2b: Multi-Party SMC Protocol

**New package:** `goldenmatch/pprl/`

**Architecture -- two computation parties + orchestrator:**

This uses a semi-honest 2PC (two-party computation) model via secret sharing. The "coordinator" is a non-computing orchestrator that sets up connections and collects the final binary output (match/no-match per pair). It never receives secret shares or intermediate values.

**Protocol -- secret-shared dice similarity:**

1. Both parties agree on schema config (field order, n-gram size, filter size, threshold)
2. Each party locally computes bloom filters with their own secret HMAC key
3. Parties engage in a 2PC protocol using SPDZ (secret sharing over arithmetic circuits via `mp-spdz`):
   a. Each party secret-shares their bloom filter bits
   b. The circuit computes: intersection = sum(a_i AND b_i), size_a = sum(a_i), size_b = sum(b_i)
   c. The circuit computes: dice = 2 * intersection / (size_a + size_b)
   d. The circuit computes: match = (dice >= threshold) -- comparison inside the circuit
   e. Only the single match bit per pair is revealed to the coordinator
4. Coordinator receives match bits, runs clustering (Union-Find), returns cluster IDs to both parties
5. Each party learns which of their records matched, but not the other party's data or bloom filters

**Threat model:** Semi-honest (honest-but-curious). Parties follow the protocol but may try to infer information from messages they receive. Malicious security (where parties may deviate from the protocol) is out of scope for v1.0.

**Crypto library:** `mp-spdz` using the SPDZ protocol family (pre-computed Beaver triples for multiplication). Requires a setup phase. Heavy dependency -- behind `pip install goldenmatch[pprl]`.

**Trusted third party fallback:** `protocol: trusted_third_party` is a thin orchestration layer over existing bloom filter + dice/jaccard code. Both parties send encrypted bloom filters to the coordinator (GoldenMatch instance), which computes similarity directly using the existing scorer. This is essentially the current PPRL pipeline with a network transport layer. Ships as the default; `smc` is the opt-in hardened mode.

**CLI interface:**

`pprl` is a Typer sub-app (like `config`), registered in `cli/main.py`:

```bash
# Party A
goldenmatch pprl serve --role party --port 9001 --data my_data.csv --config pprl.yaml

# Party B
goldenmatch pprl serve --role party --port 9002 --data their_data.csv --config pprl.yaml

# Coordinator
goldenmatch pprl link --parties localhost:9001,localhost:9002 --config pprl.yaml --output clusters.csv
```

**Config:**

```yaml
pprl:
  enabled: true
  security_level: high
  bloom_filter_size: 1024
  hash_functions: 30
  ngram_size: 2
  fields: [first_name, last_name, dob, zip]
  threshold: 0.85
  protocol: smc          # "smc" (full) or "trusted_third_party" (simpler)
```

**Scope boundary:** The protocol handles linkage only -- returns cluster IDs. Record merging / golden record creation happens on each party's side with their own data. GoldenMatch doesn't broker actual data exchange.

**Install:**

```bash
pip install goldenmatch[pprl]    # pulls mp-spdz + crypto deps
```

### Test Strategy

- **Phase 2a:** Unit tests for each bloom filter improvement. Accuracy tests on synthetic name+address dataset with known ground truth. Measure FPR/FNR at each security level against targets.
- **Phase 2b trusted_third_party:** Integration test with two localhost processes exchanging bloom filters. Verify cluster IDs match a known ground truth.
- **Phase 2b SMC:** Integration test spawning 2 `mp-spdz` processes on localhost. Verify match bits are correct for a small test dataset (100 records per party). This test is slow (~30s) so marked with `@pytest.mark.slow` and excluded from default CI. Runs in a separate CI job on merge to main only.

## Phase 3: 1.0 Freeze (v1.0.0)

No new features. Quality and stability only.

### API Freeze

- All public function signatures, CLI flags, config YAML keys, REST endpoints, and MCP tools are locked
- Document the public API surface in `docs/api-stability.md` -- explicit list of what's covered by semver
- Deprecation policy: deprecated features get a warning for one minor version before removal

### Classifier Bump

- `pyproject.toml`: `"Development Status :: 4 - Beta"` -> `"Development Status :: 5 - Production/Stable"`

### CHANGELOG.md

- Retroactive entries for every version from 0.3.0 through 0.6.0
- Going forward, every PR must update the changelog

### Documentation Sweep

- README, wiki, and docstrings all consistent with final API
- Migration guide from 0.3.x -> 1.0 if any breaking changes landed in Phase 0

### Performance Regression Tests

- Establish baseline benchmarks in CI: 1K and 10K record exact+fuzzy pipeline times
- Fail CI if runtime exceeds 2x baseline (catches accidental O(n^2) regressions)

### Release

- Tag v1.0.0, GitHub release with full changelog
- PyPI publish
- Announcement in Discussions

## Dependencies Between Phases

```
Phase 0 (CI/CD + API audit)
    |
    v
Phase 1 (LLM clustering + uncertainty)
    |
    v
Phase 2 (PPRL)
    |
    v
Phase 3 (1.0 freeze)
```

Phases are sequential. Each produces a shippable minor version (0.4, 0.5, 0.6) so users get value incrementally. Phase 3 is the final polish pass before cutting 1.0.

## What's NOT in v1.0

- Web UI (potential v1.1+)
- Hosted/cloud version
- Additional enterprise connectors beyond existing 5
- dbt materialization plugin (stays as separate package)
- Malicious-security PPRL (semi-honest only in v1.0)
